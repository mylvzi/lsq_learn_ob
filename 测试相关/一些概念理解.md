# 1.提测
	简单理解:开发人员完成开发工作,且完成相应的自测任务,交由测试人员进行更加详细,全面的测试任务
**提测（Test Submission**是软件开发流程中的关键环节，指开发团队将完成编码的功能模块、系统版本或修复的缺陷（Bug）提交给测试团队进行验证的过程。其核心目的是通过系统化测试，确保代码质量符合预期，为后续上线或发布提供可靠依据。

什么时候会产生提测呢:
1. 一个模块开发完毕
2. 系统版本的更迭
3. 某个bug修复之后测试人员重新进行bug复测

提测的流程:
1. 开发人员完成自测,并生成可测试版本(打包成安装包/部署到测试环境)
2. 提交提测申请(提测模块/功能名称-测试环境地址--测试数据说明--关联的需求)
3. 测试团队接收与准备(收到提测申请--编写测试计划与测试用例--搭建测试环境)
4. 执行测试
5. 反馈

提测的关键要素:
1. 测试环境:明确在什么环境下进行测试,关于环境需要注意两点;1是测试的环境要尽可能的和生产环境保持一致,模拟真实环境下的操作;2是要独立于开发环境,避免环境污染(也就是测试环境不能影响实际的开发环境)
2. 提测标准(entry criteria):也就是开发人员发起提测前必须要完成的一系列任务;首先是代码必须通过编译,没有严重的错误;自测任务[ to do](谁提供)要完成;关键的bug要修复
3. 提测物:要进行提测需要的物质准备,包括三方面:可执行文件,部署文档,测试数据

# 2.敏捷开发和devops实践
devops实践:dev开发 ops运维。所谓的devops实践就是将开发和运维集成在一起,不再隔离

传统方式:师傅揉面(开发)--徒弟烤面包(部署)--店员卖面包(运行)--客户反应面包太硬(bug出现)
一旦出现bug,店员需要道歉,赔偿,还需要和徒弟,师傅沟通,协作成本大

devops实践:师傅揉面(开发)--机器自动完成部署--机器自动完成测试(对于硬的面包直接退回,不参与销售)

# 3.CI/CD
**CI(continuous integration)持续集成**:开发人员每完成一部分代码就提交到git,自动触发构建,单元测试和代码扫描
**CD(continuous delivery/deployment)持续交付/持续部署**:集成完毕之后自动完成部署的工作,可以自动部署到生产环境

# 4.prd和story卡片
PRD(product requirements documents)产品需求文档,从宏观上对于整个项目进行掌控,明确各个模块的功能,实现目标

story卡片:基于**user story(用户故事)**,是对具体需求的详细描述;根据prd中的某一个需求,QA完成完备的测试

# 5.feature(功能/特性)
在测试领域中feature就是一个**独立的,可测试的功能点**,比如用户登录,购买商品等等;映射于prd或者用户故事
分类:
1. 功能性feature
2. 非功能型feature:性能,安全性,兼容性(比如是否支持https,效率是否提升)
3. 业务规则feature:某一个具体业务的执行逻辑/判断逻辑(优惠券使用条件)

# 6.随版/非随版
区别在于要测试的内容跟**版本是否有关**
如果跟版本有关(比如新版本增加的功能,新增一个支付宝支付)就是随版测试
如果跟版本无关(测试基础功能)就是非随版测试

# 7.自动化测试能代替人吗
如何理解自动化测试能代替人?
1.代码也是人写的
2.测试用例也和人有关,需要不断调整
