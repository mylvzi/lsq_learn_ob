# 核心知识
## 1.HTTP底层封装的是TCP协议
HTTP协议是一个应用层协议, 全称是Hyper Text Transfer Protocal, 即超文本传输协议

## 2.HTTP核心关注点
> HTTP协议是一个基于TCP协议的, 无状态, 请求-响应式协议

* 基于TCP : 有三次握手, 需要建立和断开连接
* 无状态 : 服务器不会保存客户的信息(引入一些额外的机制保存, 如session)
* 请求-响应式 : 永远是客户端先发起

> HTTP-Header

```css
> GET / HTTP/1.1
> Host: www.baidu.com
> User-Agent: curl/7.68.0
> Accept: *
> 
> 
< HTTP/1.1 200 OK
< Cache-Control: private, no-cache, no-store, proxy-revalidate, no-transform
< Content-Length: 2381
< Content-Type: text/html
< Pragma: no-cache
< Server: bfe
< Set-Cookie: BDORZ=27315; max-age=86400; domain=.baidu.com; path=/
< Date: Wed, 28 Jan 2026 15:50:07 GMT
```

`Host` : HTTP请求中, 告诉服务器**你要访问哪个服务器**的核心字段
		IP地址 -- 负载均衡服务器 -- 多个域名映射一个IP地址
`User-Agent` : 客户端代理是谁; 是基于浏览器还是curl命令  curl -v www.baidu.com
			可以做灰度,风控
`Content-Type` : 内容类型  text/html
`Content-Length` : body的长度, TCP是面向字节流传输, 需要知道哪里的body的结尾, 否则会出现粘包
`Connection` : 连接方式. HTTP1.1默认就是keep-alive, 即长连接, 长连接本质上是`用空间换时间`, 用带宽换取TCP断开和连接带来的时间消耗, 核心区别在于长连接`可以复用一次TCP连接`
`Cached-Control` : HTTP性能的关键, 是否支持缓存, 缓存大小是多少
`Transfer-Encoding` : 传输控制

> HTTP版本升级

**HTTP1.0 -->HTTP1.1**
1. 性能提升 : HTTP1.1默认是长连接, connection : keep-alive, 支持复用同一套TCP链接
2. 传输优化 : HTTP1.0的Header中, 必须有content-length, 以解决粘包问题, 对于动态的数据/实时数据流, 是无法应用上面的传输方式的(比如全部写完, 一次性发过去), 而HTTP1.1引入chunked, 对动态数据流, 每产生一段数据, 就将这段数据以特定的16进制格式进行**打包**, 在这个格式中有对于这个数据元数据的描述, 然后发送给前端, 可以缩短TTFB
3. 强制引入HOST, 支持虚拟主机
