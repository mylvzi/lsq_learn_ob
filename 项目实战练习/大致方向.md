# 阶段 1：MySQL + Spring Boot（业务与数据建模）

## 项目 1：用户 & 订单管理系统（单体）

### 场景（非常真实）

一个最基础的系统：

- 用户注册 / 登录
    
- 用户下单
    
- 查询订单
    

### 技术使用

- Spring Boot
    
- MySQL
    
- JPA 或 MyBatis（推荐 MyBatis，更贴近底层）
    

### 你要重点学的不是 CRUD，而是👇

#### 1️⃣ 数据建模（核心）

你要**自己设计表结构**：

- user 表
    
- order 表
    
- order_item 表（可选）
    

关注点：

- 主键设计
    
- 索引设计
    
- 外键要不要用（强烈建议：**不用**）
    

#### 2️⃣ 事务

- 下单必须是一个事务
    
- 中途失败要回滚
    

#### 3️⃣ 分层

```
controller
service
repository(mapper)
```

> ✔️ 完成标准
> 
> - 能画出 ER 图
>     
> - 知道为什么某些字段要建索引
>     
> - 能解释一次下单的事务边界
>

# 阶段 2：引入 Redis（性能 & 状态）

## 项目 2：订单系统 v2（加缓存）

### 新需求

1. 查询订单列表很慢
    
2. 用户登录状态需要保存
    

### 技术加入

- Redis
    
- Spring Data Redis
    

---

## 你要在这个项目里**刻意练的点**

### 1️⃣ Redis 用在哪里？

不是随便用，而是：

- 用户登录态（token → userId）
    
- 热点订单查询缓存
    

### 2️⃣ 缓存策略（非常重要）

你要**亲手踩坑**：

- Cache Aside
    
- 缓存穿透
    
- 缓存失效
    

举例任务：

- 订单更新后，如何保证缓存不脏？
    
- Redis 挂了系统还能不能跑？
    

### 3️⃣ Redis ≠ MySQL

你要明确：

- Redis 是**缓存 / 状态**
    
- MySQL 是**事实数据源**
    

> ✔️ 完成标准
> 
> - 能解释「为什么不用 Redis 存订单真数据」
>     
> - 能写出「先查 Redis，没命中再查 DB」的逻辑
>     
> - 能模拟 Redis 挂掉的情况
>

# 阶段 3：RabbitMQ（异步 + 解耦）

## 项目 3：下单流程异步化

### 新需求（真实业务）

- 下单成功后要：
    
    - 发通知
        
    - 写日志
        
    - 更新统计数据
        

👉 **不能影响下单响应时间**

### 技术加入

- RabbitMQ
    
- Spring AMQP
    

---

## 你要练的不是“会发消息”，而是：

### 1️⃣ 为什么要 MQ

你要在代码层感受到：

- 同步调用有多恶心
    
- 异步解耦有多舒服
    

### 2️⃣ 消息可靠性

要你处理：

- 消息丢失
    
- 重复消费（幂等）
    

### 3️⃣ 业务设计

- 下单成功 → 发消息
    
- 消费者失败怎么办？
    

> ✔️ 完成标准
> 
> - 下单接口明显更快
>     
> - 能解释「为什么 MQ 能削峰」
>     
> - 能处理重复消费问题
>
